# Phase 4 — UI Design Contract

> Visual and interaction contract for the light/dark theming foundation. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This contract formalises the design decisions locked in `04-CONTEXT.md` (D-01 through D-16) into the canonical visual reference for the planner and executor. **This is a Hugo static site with no JS framework**, so the design system is the existing single `style.css` file plus the existing CSS custom-property tokens; no shadcn gate applies.

## Typography

The codebase uses 7 distinct font sizes across 17px root + 4 weights. Phase 4 adds **zero new sizes or weights** — the toggle button reuses `.site-nav a` typography exactly (D-08).

| Role | Size | Weight | Line Height | Used For |
|------|------|--------|-------------|----------|
| Body | 1rem (17px) | 400 | 1.7 | Default body, `.page-content` |
| Nav / Toggle | 0.95rem (~16px) | 400 | 1.7 (inherited) | `.site-nav a`, **`.theme-toggle` (Phase 4)** |
| Post summary | 0.95rem (~16px) | 400 | 1.6 | `.post-item-summary` |
| Date / Footer | 0.82rem (~14px) | 400 | inherit | `.post-item-date`, `.site-footer` |
| Site title | 1.1rem (~19px) | 600 | inherit | `.site-title a` |
| Section h3 | 1.1rem (~19px) | 600 | inherit | `.page-content h3` |
| Section h2 | 1.3rem (~22px) | 600 | inherit | `.page-content h2` |
| Page title | 1.6rem (~27px) | 700 | 1.3 | `.page-title` |

**Weights:** 400 (regular), 600 (semibold), 700 (bold) already in use. Phase 4 toggle uses 400 (regular).

**Phase 4 additions:** None. The `.theme-toggle` button MUST inherit `.site-nav a` typography (D-08) — `font-size: 0.95rem`, `color: var(--text-secondary)`, no underline. Browser default `<button>` typography MUST be overridden so the button blends with the nav links visually.

**Label content** (D-05): the visible text inside the button is the literal word `Dark` (when current theme is light) or `Light` (when current theme is dark). (Color/background transition is governed by D-09; the text-content swap is structural and not animated, consistent with the project's minimalist no-framework constraint.)

### Theme-color meta sync (D-15)

`<meta name="theme-color">` content attribute MUST exactly match the active `--bg`:

| Theme | theme-color value |
|-------|-------------------|
| Light | `#FFFCF0` |
| Dark | `#100F0F` |

The IIFE sets the initial value at first paint (D-11); the toggle's click handler updates it atomically with the four other side-effects (data-theme flip, aria-pressed flip, localStorage write, theme-color update).

## Component Inventory

### `.theme-toggle` (NEW)

**Element:**

```html
<button type="button" class="theme-toggle" aria-pressed="false">Dark</button>
```

(`aria-pressed="false"` and label `Dark` shown when active theme is light. `aria-pressed="true"` and label `Light` shown when active theme is dark — semantically: dark mode is engaged when pressed, per D-07.)

**Insertion site:** inside `<nav class="site-nav">` in `themes/minimal/layouts/partials/header.html`, **after** the closing `{{ end }}` of the `range .Site.Menus.main` loop (D-06). Becomes the last child of `.site-nav`.

**States:**

| State | Trigger | Visual |
|-------|---------|--------|
| Resting (light theme) | Default | text `Dark`, `color: var(--text-secondary)` (`#6F6E69`), no underline, no border, transparent bg |
| Resting (dark theme) | After toggle to dark | text `Light`, `color: var(--text-secondary)` (`#878580` — dark-token value), same transparent bg |
| Hover | `:hover` | `color: var(--accent)` (matches `.site-nav a:hover`) |
| Focus-visible | `:focus-visible` | `outline: 2px solid var(--accent); outline-offset: 2px;` (see Focus-visible Policy below) |
| Active / pressed | `:active` | inherits browser default; no custom rule |
| Reduced-motion | `@media (prefers-reduced-motion: reduce)` | identical visuals; only the body `transition` rule is omitted (D-10) |

**Button reset rules** (required since browser-default `<button>` carries border, padding, background-color, font):

```css
.theme-toggle {
  /* match .site-nav a typography (D-08) */
  font: inherit;
  font-size: 0.95rem;
  color: var(--text-secondary);
  /* button reset */
  background: transparent;
  border: 0;
  padding: 0;
  cursor: pointer;
}

.theme-toggle:hover {
  color: var(--accent);
}
```

**Mobile (≤600px) layout:** the existing `@media (max-width: 600px) { .site-header { flex-direction: column; gap: 0.5rem; } }` rule (line 261) makes `.site-nav` stack below `.site-title`. The toggle button stacks naturally as the last child of `.site-nav`. **No new mobile rule needed.** Confirm no horizontal-scroll regression on iPhone SE (375px) viewport.

### Focus-visible Policy (NEW, scope-creep with intentional cap)

Per D-08 / Claude's-discretion bullet 5, the toggle SHOULD get a focus-visible outline. To keep keyboard accessibility consistent across the nav (rather than the toggle being the only focusable element with a visible focus ring), Phase 4 also adds the same focus-visible rule to `.site-nav a`:

```css
.site-nav a:focus-visible,
.theme-toggle:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
  border-radius: 2px;
}
```

This is a deliberate, minimal expansion (a single CSS rule with two selectors). It is NOT a full nav focus-rework and does NOT touch `.site-title a`, `.page-content a`, `.post-item-title a`, or footer links — those keep browser-default focus behaviour.

### Body transition rule (NEW)

Per D-09 / D-10, the body and theme-affected children get a 150ms ease transition on three specific properties only, gated by reduced-motion:

```css
@media (prefers-reduced-motion: no-preference) {
  body {
    transition:
      background-color 150ms ease,
      color            150ms ease,
      border-color     150ms ease;
  }
}
```

Explicitly forbidden: `transition: all`, `*` selector, transitions on layout properties (`width`, `height`, `padding`, `margin`, `transform`), or transitions outside the `prefers-reduced-motion: no-preference` media block.

| Element | Copy | Notes |
|---------|------|-------|
| Toggle label (light theme active) | `Dark` | D-05: describes target state on click |

| Empty state | not applicable | the toggle has no empty state — it always has one of two values |
| Error state | not applicable | localStorage write failures (Safari private mode) are silently swallowed via try/catch in the IIFE; theme falls back to `matchMedia` OS preference; user sees the correct theme regardless |
| Destructive confirmation | not applicable | toggling is reversible; no confirmation prompt |

**IIFE comment** (D-13): the inline `<head>` IIFE MUST contain a single-line comment at the top documenting the localStorage key name, e.g.:

```js
// Theme bootstrap — reads localStorage key 'theme', falls back to OS prefers-color-scheme.
```

The localStorage key MUST be `theme` (single, lowercase, descriptive token, matching the suggestion in D-13).

## Interaction Specification

### Toggle click handler

A single function MUST atomically perform all four side effects (per CONTEXT.md § Specifics, paragraph 2):

1. Flip `document.documentElement.dataset.theme` between `'light'` and `'dark'`.
2. Flip the toggle's `aria-pressed` attribute (`'true'` ↔ `'false'`).
3. Update the toggle's visible text content (`Dark` ↔ `Light`).
4. Write the new value to `localStorage.setItem('theme', ...)` (wrapped in try/catch for Safari private mode).
5. Update the `<meta name="theme-color">` `content` attribute to the matching `--bg` hex.

These five operations MUST live in one function and MUST NOT be split across multiple event listeners. The click handler is the single source of mutation.

### IIFE bootstrap (locked by D-11)

Runs synchronously in `<head>` BEFORE `<link rel="stylesheet">`. Required behaviours:

1. Read `localStorage.getItem('theme')` inside try/catch (Safari private mode throws on access).
2. If unset, fall back to `window.matchMedia('(prefers-color-scheme: dark)').matches`.
3. Set `document.documentElement.dataset.theme = 'dark' | 'light'` synchronously.
4. Set the initial `<meta name="theme-color">` content to match. (If the meta tag is below the IIFE in source order, query it inside `DOMContentLoaded`; if above, set it directly.)

Inline IIFE size budget: under 400 bytes after minification (CONTEXT.md § Specifics).

### Reduced-motion behaviour (D-10)

Under `prefers-reduced-motion: reduce` (or Safari's "Reduce motion" system setting), the `@media (prefers-reduced-motion: no-preference)` block does NOT load, so the `body` transition rule is absent. The theme switch is instant — no fade, no flash. This is structural, not an override.

### No-flash guarantee (THEME-02 / D-11)

Hard reload of any page in dark mode under DevTools Slow-3G MUST produce zero light frames in a screen recording (Roadmap § Phase 4 success criterion 1). Verification target: `/blog/2026-03-05-climbing-routes/` (canary page with Mermaid + Plotly + Leaflet).

## Out of Scope (DO NOT IMPLEMENT)

These are deferred or rejected — listed here so the executor cannot accidentally pull them in:

- View-transition API cross-fade on toggle (deferred to v2.x)
- Cross-tab `storage` event listener (deferred)
- Per-library reactivity for Mermaid / Plotly / Leaflet on toggle (deferred)
- Three-state light/dark/auto toggle (REJECTED, REQUIREMENTS § Out of Scope)
- `<picture>` + `prefers-color-scheme` swap for the toggle or wordmark (REJECTED — silently overrides manual toggle)
- Cookie-based persistence (REJECTED — `localStorage` only)
- PWA `site.webmanifest` with `theme_color` (deferred to v2.x — Phase 4 only adds the meta tag)
- Icons inside the toggle button (rejected by D-07 — text-only)
- Focus-visible rework on `.site-title`, `.page-content a`, `.post-item-title`, footer links (out of scope; Phase 4 cap is `.site-nav a` + `.theme-toggle` only)
- Animating the toggle's label text swap (rejected — snap is the minimal default)
- Animating any layout properties during theme switch (only `background-color`, `color`, `border-color` are allowed per D-09)
