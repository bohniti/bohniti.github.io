```
    {{ partial "footer.html" . }}
  </div>
  <script>
    // Theme toggle — single click handler performs five atomic mutations.
    (function () {
      const toggle = document.querySelector('.theme-toggle');
      if (!toggle) return;
      const meta = document.querySelector('meta[name="theme-color"]');

      // First-paint sync: server-rendered HTML always says "Dark"/aria-pressed="false".
      // If the head IIFE resolved dark, update the button to match.
      const initial = document.documentElement.dataset.theme === 'dark' ? 'dark' : 'light';
      toggle.setAttribute('aria-pressed', initial === 'dark' ? 'true' : 'false');
      toggle.textContent = initial === 'dark' ? 'Light' : 'Dark';

      toggle.addEventListener('click', function () {
        const next = document.documentElement.dataset.theme === 'dark' ? 'light' : 'dark';
        document.documentElement.dataset.theme = next;
        toggle.setAttribute('aria-pressed', next === 'dark' ? 'true' : 'false');
        toggle.textContent = next === 'dark' ? 'Light' : 'Dark';
        try { localStorage.setItem('theme', next); } catch (_) { /* Safari private mode */ }
        if (meta) meta.setAttribute('content', next === 'dark' ? '#100F0F' : '#FFFCF0');
      });
    })();
  </script>
</body>
</html>
```

Constraints (per D-05, D-07, D-13, D-15, UI-SPEC § Interaction Specification, security_threat_model T-04-06):

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User click → toggle handler → DOM/localStorage/meta | The click event itself carries no untrusted data; the handler reads only the current `data-theme` value (which it itself controls) and writes only fixed literal strings. |
| Click handler → localStorage | The handler writes only `'dark'` or `'light'` — sanitization-by-construction. No user input is concatenated into the stored value. |
| Click handler → meta tag content | The handler writes only `'#100F0F'` or `'#FFFCF0'` — sanitization-by-construction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | Tampering | localStorage write | mitigate | The click handler computes `next` as `'light'` or `'dark'` based on the current `data-theme` value, with no external input path. The two literal strings are the only values ever written. (See also Plan 02 T-04-05 — IIFE sanitizes on read.) |
| T-04-11 | Tampering | meta[name=theme-color] content | mitigate | The handler writes ONLY one of two literal hex strings (`#100F0F` / `#FFFCF0`) via setAttribute. No user input flows into the meta content. (See also Plan 02 T-04-06.) |
| T-04-12 | Spoofing | aria-pressed value | accept | A malicious browser extension could mutate aria-pressed, but the user's own AT (assistive tech) reads from the live DOM, so the handler's writes are authoritative. No mitigation needed at the app layer. |
| T-04-13 | Information Disclosure | toggle source visible in HTML | accept | Same rationale as Plan 02 T-04-07 — public-by-design. |
| T-04-14 | Denial of Service | click event flood | accept | Each click performs five cheap DOM ops; even at click-spam rates this won't degrade. No debouncing needed. |
| T-04-15 | Repudiation | theme choice attribution | accept | Theme choice is per-device (localStorage) and not tied to any account. Repudiation is meaningless here. |

Severity: low across the board. The two structurally important mitigations (T-04-10, T-04-11) are achieved by writing only literal strings, never user-controlled values.
