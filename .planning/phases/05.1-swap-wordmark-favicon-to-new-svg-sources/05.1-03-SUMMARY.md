**Why favicon.ico is essentially flat:** the ICO container packs three independent PNG-encoded frames (16/32/48), and almost all of the byte budget is the per-frame PNG header + IDAT chunks. The cairosvg-rendered raw PNG bytes happen to compress to nearly the same size as the Phase 5 PNG-of-PNG path â€” file size is dominated by the format, not the rasterization quality.

**Why apple-touch shrinks 38%:** the Phase 5 path was `favicon-light.png (RGB on solid bg) â†’ resize(180,180) â†’ pngquant`. The Plan 3 path is `cairosvg(180Ă—180, transparent bg, single fill) â†’ pngquant`. The transparent-bg PNG has far fewer distinct colors (just `#100F0F` + transparency), giving pngquant a much smaller palette to encode.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Installed libcairo system library via brew**
- **Found during:** Task 2 (first venv-run attempt)
- **Issue:** cairosvg's underlying CFFI binding (`cairocffi`) requires `libcairo.2.dylib` at runtime; macOS does not ship libcairo, and homebrew's cairo formula was not installed. The first script run failed with `OSError: no library called "cairo-2" was found`.
- **Fix:** `brew install cairo` (1.18.4, with deps libxext, libxrender, lzo, pixman, libx11, etc.) â€” mirrors the pre-existing pngquant install pattern (system binary via brew, never in `requirements.txt`). cairosvg + Pillow remain venv-only Python deps per CONTEXT D-15.
- **Files modified:** None (system-level, no repo state)
- **Commit:** N/A â€” system-level fix, not a code change

The throwaway-venv pattern in PATTERNS.md mentions "pngquant is a system binary (assumed present via brew install pngquant)" â€” by the same logic, cairo is now an assumed present system binary on the dev machine. This is consistent with project constraints (no third-party Python deps in requirements.txt) but worth flagging for any future contributor running the script on a clean machine.

**Recommendation for future docs:** the script's pre-flight check could be extended with `from cairosvg import svg2png` failure-mode hint (`brew install cairo` if ImportError mentions libcairo). Out of scope for this plan; logging here for visibility.

### cairosvg Quirks Observed

None significant. cairosvg respected the SVG's embedded `<style>` block and used `path { fill: #100F0F; }` for the rasterized output (light-mode rule). The `@media (prefers-color-scheme: dark)` rule in the SVG has no effect at cairosvg render time â€” this is the documented and intended behavior (fallback raster outputs are theme-static; the SVG itself carries the dark @media for browsers that honor it).
