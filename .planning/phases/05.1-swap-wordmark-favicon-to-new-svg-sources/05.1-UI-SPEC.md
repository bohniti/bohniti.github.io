# Phase 05.1 — UI Design Contract

> Visual and interaction contract for swapping the Phase 5 PNG-driven wordmark + favicon implementation to native SVG sources. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope note:** This phase is a **wiring/swap**, not a visual redesign. The wordmark and favicon glyphs are already authored (`images/brand-source/{logo.svg, fav-icon.svg}`) and the existing Flexoki design system (Phase 4) is binding. This contract therefore inherits Phase 4 tokens verbatim and locks **only the surfaces this phase touches**: header `.site-title svg` and the three favicon `<link>` targets.

## Color

Phase 05.1 inherits the Phase 4 Flexoki token system verbatim. The 60/30/10 mapping below documents the **inherited** site palette so the checker can confirm 05.1 does not introduce off-token colors:

| Role | Light value | Dark value | Token | Usage |
|------|-------------|------------|-------|-------|
| Dominant (60%) | `#FFFCF0` | `#100F0F` | `var(--bg)` | Page background, header background |
| Secondary (30%) | `#F2F0E5` (light) / `#1C1B1A` (dark) | `var(--bg-secondary)` | (only on code blocks; not surfaced by 05.1) |
| Body text | `#100F0F` (light) / `#CECDC3` (dark) | `var(--text)` | Drives the inline wordmark via `currentColor` (D-02) |
| Accent (10%) | `#AF3029` (light) / `#D14D41` (dark) | `var(--accent)` | Reserved-for list below — **NOT used by the wordmark** |

**Accent reserved for** (explicit list, unchanged by 05.1):
- Post-list link color (`.post-item-title a`)
- Inline body links (`.page-content a`)
- Hover state on `.site-title a`, `.site-nav a`, `.theme-toggle`, `.site-footer a`
- Focus-visible outline on nav links and theme toggle (`outline: 2px solid var(--accent)`)
- Blockquote left border (`.page-content blockquote`)

**Phase 05.1 specific color rules:**

| Surface | Light fill | Dark fill | Mechanism |
|---------|-----------|-----------|-----------|
| Inline wordmark (`.site-title svg`) | `#100F0F` (Flexoki ink) | `#FFFCF0` (Flexoki cream) | `fill="currentColor"` on all 11 paths + CSS `color: var(--text)` — recolors transitively when `[data-theme]` flips (CONTEXT.md D-02) |
| `favicon.svg` (browser tab) | `#100F0F` | `#FFFCF0` | Embedded `<style>` + `@media (prefers-color-scheme: dark)` — favicons render outside the DOM and cannot read `[data-theme]`, so OS preference is the only signal (CONTEXT.md D-09) |
| `favicon.ico` (legacy fallback) | `#100F0F` (single fill — not theme-swappable) | same | Rasterized at 16/32/48 px from `fav-icon.svg` (CONTEXT.md D-12) |
| `apple-touch-icon.png` (iOS home) | `#100F0F` on transparent bg | same (iOS adds white frame; dark fill reads on both wallpapers) | Rasterized at 180 × 180 from `fav-icon.svg` (CONTEXT.md D-13) |

**Hardcoded hex justification:** `#100F0F` and `#FFFCF0` appear inside the favicon SVG `<style>` block and inside `build_brand_assets.py` raster pipeline. They are deliberately **not CSS variables** because:
1. `<style>` inside an SVG file referenced as `<link rel="icon">` runs in an isolated CSSOM and cannot resolve `:root` custom properties from the host document
2. `favicon.ico` and `apple-touch-icon.png` are PNG/raster — they have no concept of variables

These are the **same Flexoki light/dark `--text` values** locked in `style.css` `:root` and `:root[data-theme="dark"]`. If those tokens ever change, the SVG `<style>` block, the `.ico` raster fill, and the apple-touch raster fill must be updated in lockstep — this is a known cross-file invariant the checker should flag if violated.

**Tradeoff documented (CONTEXT.md D-09):** if a user toggles in-page theme to dark while OS is in light mode, the favicon stays light. Acceptable — matches Phase 5 behavior; unfixable without a service worker (violates the no-JS-framework constraint).

## Copywriting Contract

Phase 05.1 introduces **no new user-visible copy**. The wordmark replaces text rendering with a graphic mark; the favicon set is silent (browser-chrome only). The contract below locks the strings this phase touches:

| Element | Copy | Source |
|---------|------|--------|
| Wordmark accessibility (alt) | `<svg ... role="img" aria-label="Timo Bohnstedt"> <title>Timo Bohnstedt</title> ...` | Equivalent to Phase 5 D-03 (`alt="Timo Bohnstedt"`); inline SVG cannot use `alt`, so `<title>` element + `aria-label` on the `<svg>` carries the same a11y contract |
| Wordmark anchor target | `<a href="{{ .Site.BaseURL }}">…inline SVG…</a>` (unchanged from Phase 5) | `partials/header.html` |
| `favicon.html` `<link>` tags | Three `<link>` elements; no copy. URLs unchanged: `/favicon.svg`, `/favicon.ico`, `/apple-touch-icon.png` | CONTEXT.md "Cross-Phase Contracts" — Phase 5 D-13/D-14 unchanged |
| `apple-touch-icon` accessible name | None — iOS reads the `<title>` of the page for the home-screen label | iOS platform default |
| Theme-toggle button (Phase 4 — out of 05.1 scope but adjacent) | "Dark" / "Light" toggle text | Phase 4, unchanged |
| Build-script stdout (developer-facing, not user-facing) | Final file sizes for `favicon.ico` and `apple-touch-icon.png` printed line-by-line | CONTEXT.md D-14 (matches Phase 3 verbose convention) |

**No CTA, no empty state, no error state, no destructive confirmation** — this phase has no interactive surfaces beyond the existing wordmark anchor (linking to `/`) and the existing theme-toggle button (Phase 4 territory). The wordmark anchor's interaction contract is identical to Phase 5: clicking returns the user to the site root.

**Destructive actions in this phase** (file deletions during execution, not user-visible):
- Delete 8 sliced PNGs at `themes/minimal/static/images/brand/{logo,icon,minimum,favicon}-{light,dark}.png` (CONTEXT.md D-17)
- Delete `images/logos.png` original sprite (CONTEXT.md D-18)
- Overwrite `themes/minimal/static/favicon.svg` (replaces 77 KB Phase 5 file with ~3-5 KB native path-based SVG) (CONTEXT.md D-08, D-10)
- Overwrite `themes/minimal/static/favicon.ico` and `themes/minimal/static/apple-touch-icon.png` (regenerated via build script) (CONTEXT.md D-11, D-12, D-13)

These are repo operations, not UI flows — no in-app confirmation dialog needed. The git commit log is the audit trail.

## Interaction Contract

This section augments the standard 6-pillar template because Phase 05.1 has interaction guarantees that the planner and executor must preserve. **None of these are new behaviors** — they are inherited contracts from Phase 4 + Phase 5 that 05.1 must not regress.

| Surface | Interaction | Mechanism | Inherited from |
|---------|------------|-----------|----------------|
| Wordmark color follows theme | Click theme toggle → wordmark recolors smoothly with body color transition | `currentColor` + `var(--text)` + Phase 4 `transition: color 150ms ease` on `body` | Phase 4 (style.css lines 49-56), Phase 05.1 D-02 |
| Wordmark anchor returns home | Click wordmark → navigates to `/` | `<a href="{{ .Site.BaseURL }}">…inline SVG…</a>` (unchanged) | Phase 5 |
| Wordmark theme swap timing | Theme flip and wordmark recolor are atomic — no flicker, no FOUC | Single inline SVG (no two-image swap, no JS DOM mutation) | Phase 05.1 D-01 |
| Reduced-motion respect | `prefers-reduced-motion: reduce` → no transition on wordmark color | Inherited from Phase 4 body-transition `@media` guard | Phase 4 (style.css line 49) |
| Wordmark CLS guarantee | Layout shift on first paint < 0.1 (Lighthouse Mobile) | Explicit `width="200" height="122"` attributes on `<svg>` element | Phase 5 success criterion 2; CONTEXT.md D-04 |
| Favicon swaps with OS theme | OS dark mode → tab favicon renders cream; OS light mode → tab favicon renders ink | `@media (prefers-color-scheme: dark)` inside SVG `<style>` | Phase 05.1 D-09 |
| Favicon does NOT follow in-page toggle | User clicks theme toggle → page recolors but favicon stays at OS preference | Browser tab chrome is outside the DOM; cannot read `[data-theme]` | Phase 05.1 D-09 (acknowledged tradeoff) |
| iOS home-screen icon | Add to Home Screen → TB monogram renders (not generic web preview) | `<link rel="apple-touch-icon" href="/apple-touch-icon.png">` from `partials/favicon.html` | Phase 5 success criterion 4 |
| Focus-visible parity | Tab key reaches the wordmark anchor — focus ring matches site convention | Inherited `.site-title a:focus-visible` (none currently — wordmark anchor has no explicit focus style; matches Phase 5) | Phase 5 |

**Focus state gap noted:** the existing `.site-title a` rule (style.css lines 74-79) does not declare `:focus-visible`. Phase 5 shipped without one; Phase 05.1 inherits this gap. **Recommendation for the planner:** if the executor adds keyboard nav improvements, surface this as a follow-up — but do not block 05.1 on it.
