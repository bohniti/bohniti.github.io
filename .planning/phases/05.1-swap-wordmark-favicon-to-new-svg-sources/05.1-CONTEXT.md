## Phase Boundary

Two coordinated swap tasks that retire the PNG-driven implementation shipped in Phase 5 and migrate to native SVG sources now available at `images/brand-source/{logo.svg, fav-icon.svg}`:

1. **Wordmark swap** — Replace the two `<img>` tags in `partials/header.html` (Phase 5 D-01) with an inlined SVG sourced from `logo.svg`. All `fill="#000000"` declarations become `fill="currentColor"`; CSS `.site-title svg { color: var(--text) }` drives both themes through Phase 4's `[data-theme]`-keyed token. The Phase 5 two-image display swap and per-variant seam-mask backgrounds are deleted.

2. **Favicon swap** — Replace the 77 KB PNG-wrapped `favicon.svg` (Phase 5 D-07/D-08) with a native ~3-5 KB path-based SVG sourced from `fav-icon.svg`. Theme-aware coloring via embedded `<style>` + `@media (prefers-color-scheme: dark)`. Regenerate `favicon.ico` and `apple-touch-icon.png` from the same SVG via cairosvg + Pillow inside the rewritten build script. Delete all 8 sliced PNGs and the original `images/logos.png` sprite.

**In scope:**
- Inline SVG wordmark via Hugo `readFile`/`resources.Get` + `safeHTML`
- `fill="currentColor"` substitution in `logo.svg`
- Move `logo.svg` to `themes/minimal/static/images/brand/logo.svg`
- Recompute wordmark dimensions to 200×122 / 160×98 (preserves new SVG's 1.64:1 aspect)

- Rewrite `scripts/build_brand_assets.py` to rasterize `fav-icon.svg` via cairosvg → 16/32/48 px (for `.ico`) + 180 px (for apple-touch); pngquant the apple-touch

- Delete `images/logos.png`

**Out of scope:**
- Visual redesign of the wordmark or favicon glyph
- Multi-color SVG fills (current SVG is single-fill across wordmark + climber detail; deferred)
- New favicon sizes beyond 16/32/48 + 180

<decisions>
## Implementation Decisions

### Wordmark Embedding

- **D-01:** Wordmark is **inlined SVG**. `partials/header.html` uses Hugo's `{{ readFile "themes/minimal/static/images/brand/logo.svg" | safeHTML }}` (or `resources.Get` + `.Content` — Claude's discretion) to embed the SVG markup directly into the rendered HTML. Replaces the two `<img>` tags from Phase 5 D-01. Single asset, no two-image swap, no FOUC, no JS, no theme-driven DOM swap. HTML payload grows by ~25 KB on every page (uncached) but the two prior PNG fetches (~47 KB combined) and the seam-mask CSS rules disappear.
- **D-02:** All 11 `fill="#000000"` declarations in `logo.svg` are replaced with `fill="currentColor"` before commit. CSS rule `.site-title svg { color: var(--text); }` drives the wordmark color from Phase 4's Flexoki text token (light: `#100F0F`, dark: `#FFFCF0`). Theme toggle recolors the wordmark transitively through `var(--text)` — `[data-theme]` is honored without the wordmark reading it directly.
- **D-03:** Canonical SVG location: **`themes/minimal/static/images/brand/logo.svg`** (move from `images/brand-source/logo.svg`). Co-located with brand assets convention from Phase 3; keeps the theme self-contained for any future theme swap. `images/brand-source/` may be retained as a working/source dir or deleted post-deploy (Claude's discretion).
- **D-04:** Wordmark dimensions: **200×122 px desktop, 160×98 px mobile**. Recomputed from the new SVG's `viewBox="0 0 1024 624"` (aspect 1.64:1, vs Phase 5's 1.71:1). Width footprint preserved (200/160) so the header layout doesn't shift; height grows by 5 px to honor the true SVG aspect. Explicit `width` and `height` attributes on the inlined `<svg>` element — same CLS guarantee as Phase 5 D-03.

### Wordmark CSS Cleanup (deletions)

- **D-05:** Delete the two-image display swap rules:
  ```css
  html[data-theme="light"] .wordmark-dark { display: none; }
  html[data-theme="dark"]  .wordmark-light { display: none; }
  ```
  No longer needed — single inline SVG handles both themes via `currentColor`.
- **D-06:** Delete the per-variant seam-mask backgrounds:
  ```css
  .wordmark-light { background: #FEFEFE; }
  .wordmark-dark  { background: #000000; }
  ```
  Native SVG has transparent background — no rectangular seam. **Phase 4 D-02's hard cross-phase contract is dissolved by this phase**: the corner-pixel-vs-`--bg` mismatch problem ceases to exist for SVG paths. Future phases keying off Phase 4 D-02 should treat 05.1 as the new source of truth.
- **D-07:** Mobile size override under `@media (max-width: 600px)` rewritten:
  ```css
  @media (max-width: 600px) {
    .site-title svg { width: 160px; height: 98px; }
  }
  ```
  Replaces the prior `.wordmark { width: 160px; height: 93px; }` rule (Phase 5 line ~326).

### Favicon SVG (theme-aware)

- **D-08:** `themes/minimal/static/favicon.svg` is rewritten as a **native path-based SVG** sourced from `images/brand-source/fav-icon.svg`. Replaces the 77 KB PNG-wrapped construction from Phase 5 D-07/D-08. Final size ~3-5 KB.
- **D-09:** Theme-aware coloring via **embedded `<style>` + `@media (prefers-color-scheme: dark)`**:
  ```svg
  <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 896 880">
    <style>
      path { fill: #100F0F; }
      @media (prefers-color-scheme: dark) {
        path { fill: #FFFCF0; }
      }
    </style>
    <path d="..."/><path d="..."/><path d="..."/>
  </svg>
  ```
  Favicons render in browser tab chrome outside the DOM; they cannot read `[data-theme]`. `prefers-color-scheme` is the only theme-aware selector available. **Tradeoff:** if the user toggles the in-page theme to dark while OS is in light mode, the favicon stays light — same constraint as Phase 5 D-07. Acceptable; unfixable without service-worker tricks that violate the project's no-JS-framework constraint.
- **D-10:** Source SVG at `images/brand-source/fav-icon.svg` is modified in-place to add the `<style>` block, then the modified file is copied to `themes/minimal/static/favicon.svg` as the deploy artifact. The Phase 5 favicon SVG (77 KB PNG-wrapped) is overwritten.

### Favicon `.ico` and Apple Touch Icon (rasterized from SVG)

- **D-11:** Both legacy assets are **regenerated from `fav-icon.svg`** during a build script run, using cairosvg for vector→raster conversion and Pillow for `.ico` packaging.
- **D-12:** `favicon.ico` source: rasterize `fav-icon.svg` at 16/32/48 px via cairosvg, combine into a multi-frame `.ico` via `Image.save(format="ICO", sizes=[(16,16),(32,32),(48,48)])`. Single fill (`#100F0F` Flexoki ink) — `.ico` is fallback-only and doesn't theme-swap; light fill reads cleanly against most browser tab chromes and the Windows shortcut shelf.
- **D-13:** `apple-touch-icon.png` source: rasterize `fav-icon.svg` at 180×180 via cairosvg with the same `#100F0F` fill (transparent bg), then run pngquant within the `--quality=65-90` envelope inherited from Phase 3 D-06. iOS adds its own white background frame to apple-touch icons by default, so a transparent-bg PNG with dark fill reads on both light and dark home-screen wallpapers.

### Build Script Rewrite

- **D-14:** `scripts/build_brand_assets.py` is **fully rewritten**. Slicing logic (Phase 3) and PNG-wrapped-SVG generation (Phase 5) are deleted. New responsibilities:
  - Read `images/brand-source/fav-icon.svg`
  - Rasterize via cairosvg to 16/32/48 px PNGs (in-memory)
  - Pack into `themes/minimal/static/favicon.ico` via Pillow `.save(format="ICO", sizes=[...])`
  - Rasterize via cairosvg to 180×180 PNG, run pngquant subprocess, write to `themes/minimal/static/apple-touch-icon.png`
  - Print final file sizes for both outputs to stdout (matches Phase 3's verbose convention)
  - **Do NOT generate `themes/minimal/static/favicon.svg`** — that file is hand-edited from `fav-icon.svg` (the SVG transformation is small enough to do by editing, not by generating)
- **D-15:** Throwaway-venv invocation pattern is preserved (Phase 3 + Phase 5). Adds **cairosvg** as a new venv-only dependency alongside Pillow + pngquant. Never enters `requirements.txt` (no runtime deps in this project per PROJECT.md Constraints).
- **D-16:** Conventions from `.planning/codebase/CONVENTIONS.md` § Python Script Conventions are preserved across the rewrite: PEP 8, `snake_case`, `pathlib.Path`, no type hints, single-line module docstring, `if __name__ == "__main__":` guard, `#!/usr/bin/env python3` shebang.

### Asset Cleanup

- **D-17:** Delete all 8 sliced PNGs at `themes/minimal/static/images/brand/`:
  - `logo-{light,dark}.png` — replaced by inline SVG
  - `icon-{light,dark}.png` — never wired in v2.0
  - `minimum-{light,dark}.png` — never wired in v2.0
  - `favicon-{light,dark}.png` — superseded by SVG-driven `.ico` and apple-touch
  - Saves ~163 KB of repo weight; eliminates "are these still used?" ambiguity for future phases.
- **D-18:** Delete `images/logos.png` (the original 2×4 sprite from Phase 3). No longer consumed by the build script (slicing logic removed). Brand source of truth migrates entirely to `images/brand-source/{logo.svg, fav-icon.svg}` (or to the theme-static destinations after D-03/D-10).

### Claude's Discretion

- Exact Hugo template idiom for inlining: `{{ readFile "themes/minimal/static/images/brand/logo.svg" | safeHTML }}` vs `{{ $svg := resources.Get "images/brand/logo.svg" }}{{ $svg.Content | safeHTML }}` — both produce identical output. `resources.Get` is more idiomatic Hugo but adds a Hugo Pipes dependency for a one-line read. Pick whichever reads cleaner with the existing template body.
- Whether to also pipe the inline SVG through Hugo's `minify` (`| minify | safeHTML`) — small HTML size win on every page; cost is one extra template function. Recommend yes if the source SVG retains `<defs>`/comments after the in-place fill rewrite.
- Whether `images/brand-source/` is retained after the SVG move/edit, or deleted as part of D-03/D-10 — both are reasonable. Keeping it makes future SVG edits more discoverable; deleting it makes the repo cleaner.
- Exact `<style>` placement inside the favicon SVG — class-based (`<path class="d"/>` with `.d{fill:...}`) vs path-element-targeted (`path{fill:...}`). Both render identically; path-element-targeted is shorter.
- pngquant settings for the 180×180 apple-touch — within the `--quality=65-90` envelope from Phase 3 D-06.
- Whether to delete the existing `favicon.ico` and `apple-touch-icon.png` files first then let the build script produce fresh files, or overwrite in place. Either works; deleting first surfaces missing-file errors loudly if the build script regression tests fail.
- Whether to verify the build script run actually produces the expected outputs before committing (smoke-test by inspecting file sizes and running `file favicon.ico` for multi-frame validation). Recommend yes — Phase 5's HUMAN-UAT pattern flags real validation as a gate.

### Folded Todos

None — no pending todos in STATE.md matched this phase's scope.
