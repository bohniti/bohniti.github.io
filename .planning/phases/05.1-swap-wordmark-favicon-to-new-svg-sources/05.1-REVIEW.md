# Phase 05.1: Code Review Report

**Reviewed:** 2026-04-29
**Depth:** standard
**Files Reviewed:** 6
**Status:** warnings

## Summary

Phase 05.1 swaps the legacy `<img>`-based wordmark and PNG favicon for inlined-SVG sources driven by `currentColor` and a new `cairosvg`-based asset pipeline. The changes are small in surface area and well-scoped:

- `build_brand_assets.py` was rewritten from a Pillow-only PNG export into a `cairosvg + Pillow + pngquant` rasterizer that emits `favicon.ico` (16/32/48) and `apple-touch-icon.png` (180x180) from `images/brand-source/fav-icon.svg`. The script is short, follows project Python conventions (snake_case, `pathlib.Path`, single-line module docstring, `__main__` guard, no type hints), and uses `subprocess.run` safely with a list-form argument and `check=True`.
- `partials/header.html` now inlines the wordmark via `readFile | safeHTML`. The path target exists (`themes/minimal/static/images/brand/logo.svg`, 25 KB), and a manual security scan of the SVG (next bullet) found no script-injection vectors, so `safeHTML` here is acceptable for a repo-committed asset.
- `style.css` cleanly retires the entire `.wordmark*` class family (verified: zero residual references in `themes/minimal/`) and replaces them with a single `.site-title svg` rule using `var(--text)` plus a mobile override at 600px.
- All three SVGs are clean: no `<script>`, no `<foreignObject>`, no `on*=` event handlers, no `javascript:` URIs, no `<use href>` external references. `xmllint` confirms well-formed XML, and `themes/minimal/static/favicon.svg` is byte-identical to `images/brand-source/fav-icon.svg`.

Two `Warning`-level items are worth addressing before close-out: the `logo.svg` `width`/`height` attributes are stale relative to the CSS-declared sizing and bypass `currentColor` recoloring on browsers that respect intrinsic SVG dimensions, and the script swallows `cairosvg`/`Pillow` exceptions with a raw traceback rather than the same friendly `sys.exit` path used for `pngquant`. The `Info` findings are minor consistency notes.

No critical issues. No security issues. No source modifications required for correctness.

## Warnings

### WR-01: Inlined `logo.svg` ships with hardcoded `width="200" height="122"` that duplicates and can desync from CSS sizing

**File:** `themes/minimal/static/images/brand/logo.svg:1`
**Issue:**
The SVG root carries `width="200" height="122"` as XML attributes:

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 1024 624" width="200" height="122" role="img" aria-label="Timo Bohnstedt">
```

These are duplicated (and overridden) by `.site-title svg { width: 200px; height: 122px; }` in `style.css:128-129`, with a separate mobile override at `style.css:319` (`width: 160px; height: 98px`). Two things to flag:

1. **Source of truth duplication.** Any future resize requires editing both the SVG attributes and two CSS rules. The CSS is the responsive winner today, but the inline attributes will silently become stale (and they already disagree with the mobile breakpoint).
2. **`currentColor` interaction is fine here, but only because `viewBox` is present.** With `viewBox` set, the `width`/`height` attributes are layout-only and don't break the `fill="currentColor"` recolor pattern. Worth noting because the design contract for this phase is "SVG inherits `var(--text)` via `currentColor`" — keeping intrinsic dimensions on the source mixes layout concerns with the brand asset.

**Fix:**
Drop the layout attributes from the SVG and let CSS own all sizing. This keeps `style.css` as the single source of truth for header sizing across breakpoints:

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 1024 624" role="img" aria-label="Timo Bohnstedt">
```

The CSS rules at `style.css:125-130` and `style.css:319` already define `width`/`height`, so visual output is unchanged.

### WR-02: `build_brand_assets.py` only handles missing-`pngquant` and missing-source errors; rasterization failures escape as raw tracebacks

**File:** `scripts/build_brand_assets.py:33-44, 47-65`
**Issue:**
`main()` checks `shutil.which("pngquant")` and `FAV_ICON_SVG.exists()` and exits cleanly on both, but `build_favicon_ico` and `build_apple_touch` will surface a raw `cairosvg.Error`, `PIL.UnidentifiedImageError`, `PermissionError`, or `subprocess.CalledProcessError` from `run_pngquant` directly to the user. For a one-shot reproducibility script that's run rarely, the asymmetry is minor — but it's inconsistent with the friendly handling already in place at lines 48-51.

In particular, `run_pngquant` (line 25) does `subprocess.run(..., check=True)` without capturing stderr, so a `pngquant` failure dumps both a Python traceback *and* whatever pngquant printed, which is noisy.

**Fix:**
Wrap the two builder calls in `main()` and convert `CalledProcessError` to a clean `sys.exit`. Minimal change:

```python
def main():
    if shutil.which("pngquant") is None:
        sys.exit("pngquant not found - install with 'brew install pngquant' (macOS) or your package manager")
    if not FAV_ICON_SVG.exists():
        sys.exit("Source SVG not found: {}".format(FAV_ICON_SVG))

    ico_path = THEME_STATIC_DIR / "favicon.ico"
    apple_path = THEME_STATIC_DIR / "apple-touch-icon.png"

    try:
        build_favicon_ico(FAV_ICON_SVG, ico_path)
        build_apple_touch(FAV_ICON_SVG, apple_path)
    except subprocess.CalledProcessError as exc:
        sys.exit("pngquant failed (exit {}) for {}".format(exc.returncode, exc.cmd[-1]))
    except OSError as exc:
        sys.exit("Asset build failed: {}".format(exc))
    ...
```

Optional: also confirm `THEME_STATIC_DIR` exists before writing (it does today, but a defensive `THEME_STATIC_DIR.mkdir(parents=True, exist_ok=True)` makes the script portable across fresh clones).

## Info

### IN-01: `images/brand-source/fav-icon.svg` is byte-identical to `themes/minimal/static/favicon.svg` — clarify which is source

**File:** `images/brand-source/fav-icon.svg`, `themes/minimal/static/favicon.svg`
**Issue:**
`diff -q` reports no difference between the two files. `build_brand_assets.py` reads `images/brand-source/fav-icon.svg` as the rasterization source, but neither file references the other and neither is generated from the other — they're maintained as parallel copies. If a future edit lands in only one of them, the favicon and the apple-touch-icon will silently drift.

**Fix:**
Pick one canonical source and make the other a build artifact (or remove the duplication). Two reasonable options:
1. Treat `images/brand-source/fav-icon.svg` as canonical, and have `build_brand_assets.py` also `shutil.copy2` it to `themes/minimal/static/favicon.svg` so the static site always tracks the source.
2. Treat `themes/minimal/static/favicon.svg` as canonical (since it's what's served) and point the script at it directly, deleting `images/brand-source/fav-icon.svg`.

Either resolves the drift risk; option (1) is more consistent with the "brand-source is canonical" pattern already implied by the directory name.

### IN-02: `PNGQUANT_CMD` contains a flag with `=` syntax — fragile across pngquant versions

**File:** `scripts/build_brand_assets.py:21`
**Issue:**
`PNGQUANT_CMD = ["pngquant", "--quality=65-90", "--strip", "--force", "--output"]`

Using `--quality=65-90` (with `=`) works on all current pngquant releases, but pngquant's CLI also accepts `--quality 65-90` (space-separated) in newer versions and the `=` form has been the source of subtle parsing quirks historically. More importantly, the constant ends with `--output` and is concatenated with `[str(path), str(path)]` at the call site, which makes the argv shape implicit (option, value, positional input).

This is not a security issue (list form, `shell=False`), but it's slightly brittle.

**Fix:**
Either keep the current form (it works) or split for explicitness:

```python
PNGQUANT_CMD = ["pngquant", "--quality", "65-90", "--strip", "--force", "--output"]
```

Add a brief comment at the call site clarifying that the same path is passed for both `--output` and the positional input (in-place replacement):

```python
def run_pngquant(path):
    # In-place: --output <path> <path>
    subprocess.run(PNGQUANT_CMD + [str(path), str(path)], check=True)
```

### IN-03: `apple-touch-icon.png` written then re-read by pngquant — small redundancy, no correctness issue

**File:** `scripts/build_brand_assets.py:40-44`
**Issue:**
`build_apple_touch` does `out_path.write_bytes(png_bytes)` and then calls `run_pngquant(out_path)`, which reads the same file back, optimizes it, and overwrites it. This is the standard pngquant pattern and is correct, but it's worth noting that `build_favicon_ico` does *not* run pngquant on the `.ico` (which is correct — pngquant only handles PNG, and ICO multi-frame compression is a different beast). The asymmetry is intentional but would benefit from a one-line comment so a future maintainer doesn't "fix" it by adding pngquant to the ICO path.

**Fix:**
Add a comment in `build_favicon_ico`:

```python
def build_favicon_ico(svg_path, out_path):
    """Multi-frame .ico (16/32/48) rasterized from fav-icon.svg (D-12)."""
    # Note: no pngquant step — pngquant only handles PNG, not ICO containers.
    png_bytes = svg2png(url=str(svg_path), output_width=48, output_height=48)
    img = Image.open(io.BytesIO(png_bytes)).convert("RGBA")
    img.save(out_path, format="ICO", sizes=ICO_SIZES)
```

### IN-04: `partials/header.html` `readFile` path is theme-coupled — Hugo theme override breaks silently

**File:** `themes/minimal/layouts/partials/header.html:4`
**Issue:**
`{{ readFile "themes/minimal/static/images/brand/logo.svg" | safeHTML }}` hardcodes the `themes/minimal/` prefix. Hugo's `readFile` is rooted at the project working directory, not the theme, so this works today because the theme is vendored at exactly that path. If the theme is ever renamed, mounted, or extracted into a Hugo Module, this string silently fails (Hugo's `readFile` returns an empty string and logs a warning rather than erroring out, depending on Hugo version).

This is a known sharp edge of `readFile` vs. `resources.Get`; not a bug today, just a future-coupling note.

**Fix (optional, low-priority):**
If you want to stay future-proof, switch to a resource lookup that respects theme overrides:

```go-html-template
{{ with resources.Get "images/brand/logo.svg" }}
  {{ .Content | safeHTML }}
{{ end }}
```

This reads from `assets/` (or `static/` via Hugo Pipes in newer Hugo) and survives theme reorganization. Today's `readFile` form is fine for a single-theme single-author site — flagged purely as a forward-compatibility note.
