## Typography

The gallery page uses the existing `.page-title` rule for its `<h1>` and emits **no body copy** (no description prose, no per-photo captions). The grid is the page content.

| Role | Size | Weight | Line Height | Source |
|------|------|--------|-------------|--------|
| Page heading (`<h1>`) | `1.6rem` (~25.6px desktop / ~24px mobile) | `700` | `1.3` | existing `.page-title` rule, style.css lines 174–180 — UNCHANGED |
| Body / secondary text | not used on gallery page | n/a | n/a | (gallery emits zero non-heading text) |
| `aria-label` (screen-reader-only on `<a>` wrappers) | n/a (not visually rendered) | n/a | n/a | screen-reader semantics only — see § Copywriting Contract |

**Font family declarations:**
- Inherits from `body` (system stack, style.css line 42) — no per-gallery font override
- No webfonts loaded; matches PROJECT.md "no `@font-face`" constraint

**Heading inclusion decision (CONTEXT.md "Claude's Discretion"):**
- **Decision: INCLUDE `<h1 class="page-title">{{ .Title }}</h1>`** wrapped in `<div class="page-header">`.
- **Rationale:** SEO + nav-target context (the URL `/gallery/` becomes meaningful when crawled), screen readers announce a page name, and the heading visually anchors the grid below it. Visual conflict with the photo grid is minimal because `.page-title` is a single-line `1.6rem` heading with `2rem` bottom-margin breathing room before the grid.
- **Alternative considered (rejected):** skip the heading entirely — visually cleaner but degrades a11y/SEO/landmark navigation. Cost > benefit.

**Mobile typography (under existing `@media (max-width: 600px)`):**
- `html { font-size: 16px }` (existing rule line 315) → `.page-title` becomes `1.6rem × 16px = 25.6px → ~25px` (desktop is `1.6rem × 17px = 27.2px`). No additional mobile heading override needed.

## Copywriting Contract

The gallery page is content-light by design (the photos are the content). The copywriting surface is small: a page title, screen-reader labels, and (implicit) menu copy.

| Element | Copy |
|---------|------|
| Page title (`<h1>`) | `"Gallery"` (from front-matter `title: "Gallery"`, sourced via `{{ .Title }}` in template) |
| `<title>` tag | `"Gallery — Timo Bohnstedt"` (rendered automatically by `baseof.html` line 8 logic: `{{ if not .IsHome }}{{ .Title }} — {{ end }}{{ .Site.Title }}`) |
| Menu entry (header nav) | `"Gallery"` (from `hugo.toml` `[[menu.main]] name = "Gallery"`) |
| Primary interaction (thumbnail `<a>`) | NO visible CTA text — the click target is the thumbnail itself |
| `aria-label` on each `<a class="gallery-item">` | `"Open photo {{ add $idx 1 }} of {{ $total }} at full size"` (e.g. `"Open photo 1 of 18 at full size"`, `"Open photo 12 of 18 at full size"`) |
| `alt` attribute on each `<img>` | `""` (empty — D-18, decorative; the wrapping `<a>` provides the accessible name via `aria-label`) |
| Empty-state heading | n/a — the gallery launches with all 18 photos in the bundle; an empty bundle is a build-time anomaly, not a user-facing state |
| Empty-state body | n/a — the `{{ with .Resources.Match "photos/*" }}` guard (RESEARCH § Code Excerpts) emits nothing if the bundle is empty; no visible empty-state copy is rendered |
| Error state | n/a — Hugo build fails loudly if photos are missing; users never see an error state |
| Destructive confirmation | n/a — no destructive actions on the gallery page |
| Loading state | n/a — first 3 thumbnails eager-load, remaining 15 lazy-load via native browser behavior; no loading spinners, no skeleton screens (the WebP thumbnails are small enough to populate in-place as the user scrolls) |

**aria-label wording rationale (resolves CONTEXT.md "Claude's Discretion"):**
- **Pattern: `"Open photo {{ add $idx 1 }} of {{ $total }} at full size"`** — uses `{{ $total := len . }}` outside the range loop and `{{ add $idx 1 }}` inside (1-indexed for human readability per RESEARCH).
- **Why this wording:**
  - "Open" describes the action (link, not button), more accurate than "View"
  - "photo N of M" gives positional context to a screen-reader user navigating the grid
  - "at full size" describes what happens after activation (browser navigates to the 1600×1600 fit WebP)
- **Why NOT filename-based** (e.g. `"Open IMG_1646.jpeg"`): filenames are camera/phone artifacts (DSC09782, IMG_1646), provide no semantic value to a screen reader, and would expose the kind of camera-identifying information that D-13 just stripped from EXIF.
- **Why NOT "Open photo {{ $idx }}"** (raw 0-indexed): screen readers announcing "Open photo 0" through "Open photo 17" is jarring; 1-indexed is the human-readable convention.

**`<title>` tag content for the gallery page (verified via baseof.html line 8 logic):**
- On load: `Gallery — Timo Bohnstedt` (matches site convention: page title — site title)

### Responsive Breakpoints (column count by viewport width)

Computed from RESEARCH § Responsive Collapse Math. The site has body padding `0 1.5rem` (48px total horizontal padding); content width = viewport − 48 − any wrapper-edge gutter.

| Viewport width | Effective grid width | Columns | Notes |
|----------------|---------------------|---------|-------|
| ≥ 1100 px | 1052 px (clamped to 1100 max via D-09 override) | **4 columns** | Desktop target |
| 968–1100 px | viewport − 48 px | **4 columns** | (`220×4 + 16×3 = 928 px` fits down to ~976 px viewport) |
| 728–968 px | viewport − 48 px | **3 columns** | (`220×3 + 16×2 = 692 px` fits down to ~740 px viewport) |
| 488–728 px | viewport − 48 px | **2 columns** | (`220×2 + 16 = 456 px` fits down to ~504 px viewport) |
| < 488 px | viewport − 48 px | **1 column** | Phone-default; auto-fill collapses naturally |

**Empirical correction noted in RESEARCH § Open Questions Q1:** CONTEXT.md D-11 claimed 1-column collapse below 600 px viewport — the correct threshold is ~488 px. Behavior is acceptable as-is (typical phones are 360–414 px viewport-width and already render 1 column; 2 columns at the 488–600 px boundary is a reasonable transitional layout). **No additional `@media` rule needed.**

**Existing `@media (max-width: 600px)` block stays unchanged.** The auto-fill grid handles the gallery's responsive behavior; the existing media query only adjusts root font-size, header layout, wrapper padding, footer-link gap, and wordmark size — none of which touch `.gallery-grid`.

### Element Specs

#### `.gallery-grid` (the wrapping container)

```css
.gallery-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(220px, 1fr));
  gap: 1rem;
  margin: 0;
}
```

- `display: grid` — required for grid template
- `grid-template-columns: repeat(auto-fill, minmax(220px, 1fr))` — D-08 locked verbatim. **Use `auto-fill`, NOT `auto-fit`** (RESEARCH § auto-fill vs auto-fit: with 18 items and rows of 4 columns, `auto-fit` would stretch the trailing 2 photos to span the full row; `auto-fill` keeps them at column natural width)
- `gap: 1rem` — D-08; matches the 16px md spacing token
- `margin: 0` — explicit zero override against any inherited margin from `.page-content` patterns

#### `.gallery-item` (each clickable thumbnail wrapper)

```css
.gallery-item {
  display: block;
  line-height: 0;        /* kill baseline gap below <img> */
  border-radius: 4px;
}

.gallery-item:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}
```

- `display: block` — required so the anchor fills the grid cell
- `line-height: 0` — D-08; eliminates the inline-baseline whitespace gap below `<img>` that would otherwise create a 4–6px gap inside each grid cell
- `border-radius: 4px` — matches the radius on `.gallery-img` so the focus outline traces a rounded rectangle aligned with the photo edges
- `:focus-visible` outline matches existing `.site-nav a:focus-visible` and `.theme-toggle:focus-visible` patterns (style.css lines 117–122) — same `2px solid var(--accent)` + `2px outline-offset`. **No `transition` on the outline** (existing patterns have none — match the convention)

#### `.gallery-img` (the visual thumbnail)

```css
.gallery-img {
  width: 100%;
  height: auto;
  display: block;
  border-radius: 4px;
}
```

- `width: 100%` — fills the grid cell horizontally
- `height: auto` — preserves the 600×400 (3:2) aspect ratio enforced by Hugo's `Process "fill 600x400 Smart webp q75"`
- `display: block` — eliminates inline-default baseline gap (defense-in-depth alongside `.gallery-item { line-height: 0 }`)
- `border-radius: 4px` — matches existing `.page-content img` (style.css line 225) for visual consistency across the site

### Interaction Spec

| Interaction | Behavior |
|-------------|----------|
| Mouse hover on thumbnail | Cursor changes to `pointer` (browser default for `<a href>`); **no other visual change** — no opacity dim, no scale, no overlay (locked: zero hover styling) |
| Mouse click on thumbnail | Same-tab navigation to the full-size WebP `.RelPermalink`; browser back button returns to gallery (RESEARCH-verified scroll restoration is automatic via browser session history) |
| Keyboard `Tab` / `Shift+Tab` | Focus traverses the 18 grid items in DOM order (alphabetical by filename — D-04). Focus indicator is the `:focus-visible` outline above |
| Keyboard `Enter` or `Space` | Activates the focused `<a>` (browser default); navigation behaves identically to mouse click |
| Touch tap | Same as mouse click; tap target is the entire 220px-min thumbnail (well above the 44×44px WCAG target-size minimum) |
| Browser back from full-size | Returns to gallery page; native browser scroll-restoration places the user at their prior scroll position |
| Loading first paint (above the fold) | First 3 thumbnails fetch with `loading="eager"` (D-05); first thumbnail also has `fetchpriority="high"` (D-06) to win the LCP race |
| Loading below the fold | Photos 4–18 use `loading="lazy"` + `decoding="async"` (D-05 + RESEARCH-recommended); thumbnails populate as user scrolls |
| Layout shift (CLS) | Zero. Each `<img>` has explicit `width="600" height="400"` from `$thumb.Width`/`$thumb.Height`; the `width: 100%; height: auto` CSS preserves the 3:2 aspect ratio so the browser reserves the correct cell height before the image bytes arrive |

## Body-Class Pattern Documentation

Documenting the canonical body-class pattern this phase introduces, since multiple phases will consume it:

**Signature:**
```html
<body class="page-{{ .Type | default "default" }}">
```

**Page-types this produces (verified RESEARCH § Type Quirk):**

| Page route | `.Type` | Body class | Notes |
|------------|---------|------------|-------|
| Homepage `/` | `"page"` | `page-page` | Hugo's homepage `.Type` defaults to `"page"`; the `default "default"` fallback never fires (documented surprising behavior) |
| Blog list `/blog/` | `"blog"` | `page-blog` | from section name |
| Blog post `/blog/YYYY-MM-DD-slug/` | `"blog"` | `page-blog` | leaf bundle inherits section type |
| `/gallery/` | `"gallery"` | `page-gallery` | **Phase 6's CSS hooks here** |
| `/about/` (today, `content/about.md` standalone) | `"page"` | `page-page` | will become `page-about` after Phase 7 leaf-bundle conversion |

**Implication:** The fallback `default "default"` is harmless cruft (never fires) but signals defensive intent. Future maintainers should not assume "homepage = page-default" (it's actually `page-page`). If Phase 7+ wants a dedicated homepage class, switch to `{{ if .IsHome }}home{{ else }}page-{{ .Type }}{{ end }}` (out of scope for Phase 6).
