## Spacing Scale

The project does not declare a global spacing-token scale; it uses `rem`-based ad-hoc values throughout `style.css`. For Phase 9, all spacing decisions snap to a **0.25 rem (≈ 4 px)** lattice, building on the existing About block's vocabulary (`0.75 rem`, `1.5 rem`, `1.75 rem`, `2 rem`, `2.5 rem`).

Declared spacing tokens (resolved px values calculated against the desktop root font-size, `html { font-size: 17px }` from `style.css:36`):

| Token | rem | Desktop px (17 px root) | Mobile px (16 px root, < 600 px) | Usage |
|-------|-----|-------------------------|-----------------------------------|-------|
| xs | 0.25 rem | ≈ 4 px | 4 px | Card-internal gap between role title meta line and bullets; bullet-list `li` margin |
| sm | 0.5 rem | ≈ 8 px | 8 px | Pullquote vertical padding (existing); role-card title-to-dates spacing |
| md | 1 rem | ≈ 17 px | 16 px | Card padding-x on mobile collapse; role-card paragraph margin |
| lg | 1.5 rem | ≈ 25 px | 24 px | Role-card padding (top/right/bottom/left); split-row gap; mobile reflow gap |
| xl | 2 rem | ≈ 34 px | 32 px | Hero gap (existing `.about-hero { gap: 2rem }` line 344); split-row outer margin |
| 2xl | 2.5 rem | ≈ 42 px | 40 px | Section-to-section margin (between `.about-section` blocks); existing hero `margin-bottom: 2.5rem` line 346 |
| 3xl | 3.5 rem | ≈ 60 px | 48 px | Top-of-section margin between Experience / Education / Certifications / Outside Work blocks (rhythm break — slightly larger than 2xl to read as a major divider without horizontal rules) |

**Exceptions:**
- **Pullquote internal padding** stays at the existing values (`padding: 0.5rem 0 0.5rem 1.25rem` — `style.css:363`). The 1.25 rem (~21 px) left-pad is asymmetric **on purpose** so the 4 px `border-left` reads as the visual anchor. Snapping to 1.5 rem here would weaken that anchor. Pitfall 15 contrast invariant scoped this rule and the lattice exception is part of preserving it.
- **Pullquote one-sided radius** stays at `border-radius: 0 4px 4px 0` (`style.css:366`) — the asymmetric radius IS the visual signal. Bumping to `--radius-soft` (12 px) breaks the visual relationship with the 4 px `border-left` (D-08 explicit).
- **`.page-content img` margin** at `style.css:260` (`margin: 1.5rem 0`) is global — not part of this phase. The existing `.about-grid img { margin: 0 }` cascade override at `style.css:390` stays load-bearing for any About image rendered through the render-image hook (the override comment "RESEARCH amendment 1" is preserved).
- **Mobile (< 600 px)**: every multi-column section collapses to `1fr`, gaps reduce to `1.5 rem` (lg) consistent with the existing `.about-hero` mobile rule at `style.css:439`. No new breakpoint.

## Typography

The site declares root font size at `html { font-size: 17px }` desktop, dropping to `16 px` at `< 600 px` (`style.css:36, 432`). All Phase 9 type sizes use `rem` so they scale with the root.

This phase uses **4 sizes and 2 weights** — minimum required by the 6-dimension checker.

| Role | Size | Weight | Line Height | Where used |
|------|------|--------|-------------|------------|
| Body | **1 rem** (≈ 17 px desktop / 16 px mobile) | **400** (regular) | **1.7** (inherited from `body` at `style.css:45`) | Hero intro paragraph; split-row prose; role-card paragraph; "Outside Work" prose; bullet-list items |
| Label / Meta | **0.85 rem** (≈ 14.5 px desktop / 13.6 px mobile) | **400** | **1.5** | Role-card date range (e.g. `Vienna · since 09/2023`); section sub-meta (Education degrees minor line); "(2024)" cert dates |
| Heading | **1.3 rem** (≈ 22 px desktop / 20.8 px mobile) | **600** (semibold) | **1.3** | `.about-section h2` — section labels (`Experience`, `Education`, `Certifications`, `Outside Work`); inherits the existing `.page-content h2` size at `style.css:227` so a Phase-9-specific H2 rule is **not** added unless visual review demands it |
| Display | **1.4 rem** (≈ 24 px desktop / 22.4 px mobile) | **700** (bold; `1.4 rem strong` inside the 1.4 rem aside qualifies as "large bold text" per WCAG 1.4.3) | **1.4** | `.about-pullquote` body at weight 500; **`.about-pullquote strong` at weight 700 — locked by Pitfall 15 / REQ ABOUT-06 (do not reduce size or weight)** |

**Role-card title:** **1 rem, weight 600, line-height 1.4** — same scale as body but with the 600-weight "Heading" register. Decision rationale: bumping to 1.1 rem invites visual competition with section H2s (1.3 rem). Keeping the title at body-size with bold weight signals "this is a card label, the section H2 is the section signpost." Matches Kindle/Obsidian flat hierarchy.

**Removed / not used in this phase:**
- No `font-style: italic` is added to new rules. The existing `.page-content blockquote` (`style.css:272-278`) keeps italic styling for blog blockquotes; the About pullquote uses its own non-italic rule (`style.css:357-375`) and that stays.
- No new `letter-spacing`, `text-transform`, or `font-feature-settings` declarations. Flexoki/Kindle aesthetic favors flat typography.
- No web font, no `@font-face` (FEATURES.md explicit anti-feature; PROJECT.md system-stack lock).

**Pitfall 15 contract (REQ ABOUT-06) — locked verbatim:**
```css
/* Dark-mode #D14D41 on #1C1B1A measures 3.97:1 — passes WCAG AA only because the
   inherited 1.4rem + font-weight:700 qualifies as "large text" (≥14pt bold).
   Do not reduce font-size or font-weight here without re-checking contrast. */
body.page-about .about-pullquote {
  font-size: 1.4rem;
  font-weight: 500;
  line-height: 1.4;
  /* … rest unchanged from style.css:357-367 … */
}
body.page-about .about-pullquote strong {
  color: var(--accent);
  font-weight: 700;
}
```
This block is the **only** load-bearing typography contract in the phase. The contrast comment at `style.css:369-371` MUST be preserved when the section is rewritten — it documents why the rule cannot be edited.

### Text-on-tint contrast verification (CRITICAL — REQ ABOUT-04 + REQ ABOUT-06)

All text in Phase 9 lives on either `var(--bg)` (already AA-clean from v2.0) or `var(--bg-secondary)` (new role-card surface — must pass AA). Computed ratios for every Phase-9 text-on-tint pairing:

| Pairing | Light (#text on #bg-secondary `#F2F0E5`) | Dark (#text on #bg-secondary `#1C1B1A`) | WCAG AA threshold | Passes? |
|---------|------------------------------------------|------------------------------------------|-------------------|---------|
| Body text on role card (`var(--text)` on `var(--bg-secondary)`) | `#100F0F` on `#F2F0E5` ≈ **17.5 : 1** | `#CECDC3` on `#1C1B1A` ≈ **12.4 : 1** | 4.5 : 1 (normal) | YES — both clear by > 2.5× margin |
| Meta text on role card (`var(--text-secondary)` on `var(--bg-secondary)`) | `#6F6E69` on `#F2F0E5` ≈ **4.7 : 1** | `#878580` on `#1C1B1A` ≈ **5.1 : 1** | 4.5 : 1 (normal) | YES — light is borderline (4.7 > 4.5); flagged below |
| Bullet text on role card (`var(--text)` on `var(--bg-secondary)`) | identical to body row | identical to body row | 4.5 : 1 | YES |
| Pullquote body on tint (`var(--text)` on `var(--bg-secondary)`) | identical to body row | identical to body row | 4.5 : 1 | YES |
| Pullquote `<strong>` on tint (`var(--accent)` on `var(--bg-secondary)`) | `#AF3029` on `#F2F0E5` ≈ 4.8 : 1 | **`#D14D41` on `#1C1B1A` ≈ 3.97 : 1** | 3.0 : 1 (large bold ≥ 14 pt bold) | YES (large-text path only — Pitfall 15 lock) |
| Border on role-card edge (`var(--border)` on `var(--bg-secondary)`) | `#E6E4D9` on `#F2F0E5` ≈ 1.1 : 1 | `#282726` on `#1C1B1A` ≈ 1.5 : 1 | 3.0 : 1 (UI components — WCAG 1.4.11) | **flagged** — see note below |

**[FLAG → recommendation]** Border-on-tint contrast is below WCAG 1.4.11 (`3 : 1`). This matches the existing pullquote treatment (`style.css:362` uses `1px` border via `border-left`, which is exempt because the `var(--accent)` left-stripe carries the visual weight). For role cards we apply the same Flexoki-flat strategy: the **fill contrast against the page** does the work of distinguishing the card, not the border. Verified: `var(--bg-secondary)` (`#F2F0E5`) on `var(--bg)` (`#FFFCF0`) ≈ 1.04 : 1 — also low, but the pullquote already validates this aesthetic decision visually and with users (Phase 7 deployed). **Recommendation:** Keep the `1px solid var(--border)` per D-03; do not promote to `2px` or to `var(--accent)`. WCAG 1.4.11 contrast requirement is satisfied by the **content text inside the card** (the AA-passing rows above) — the border is decorative reinforcement of an already-distinguishable surface (fill change). If a HUMAN-UAT pass surfaces "cards feel undifferentiated," upgrade to `1.5px` before changing color tokens.

**[FLAG → recommendation]** Light-mode meta text (`var(--text-secondary)` `#6F6E69` on `#F2F0E5`) is **at the cliff** at 4.7 : 1. Acceptable per AA but with no headroom. **Recommendation:** Do not put any sub-14 px text on this pairing. Meta text rendered at **0.85 rem ≈ 14.5 px** is safe. Do not drop to 0.8 rem without re-checking.

### Asymmetric ratios for `{{< split >}}` (REQ ABOUT-01, D-01)

The split shortcode is the load-bearing carrier of REQ ABOUT-01's "alternating ratios" language. Each split row auto-tracks its parent `.about-section`'s **odd/even occurrence** within the page and inverts ratio direction:

| Occurrence | Desktop columns | Mobile columns | Visual |
|------------|-----------------|----------------|--------|
| 1st split (e.g. inside Experience) | **`grid-template-columns: 2fr 1fr;`** (text 2/3, image 1/3) | `1fr` | Prose-dominant; image punctuates right |
| 2nd split (e.g. inside Outside Work intro) | **`grid-template-columns: 1fr 2fr;`** (image 1/3, text 2/3) | `1fr` (source order) | Image-led; text trails |
| 3rd split (if a 3rd is needed) | back to **`2fr 1fr`** | `1fr` | Rhythm reset |

Implementation strategy: a CSS attribute selector keyed on the shortcode's emitted `data-split-index` (0-based, parity-driven) — or, simpler, two CSS class variants (`.about-split--text-first` / `.about-split--image-first`) selected by the shortcode template based on a positional argument. **Recommendation: two-variant class approach** — readable in CSS and in the markdown, no JS or Hugo `seq` arithmetic needed.

```css
body.page-about .about-split {
  display: grid;
  gap: 2rem;                         /* xl token */
  align-items: start;
  margin: 2.5rem 0;                  /* 2xl token, between rows */
}
body.page-about .about-split--text-first {
  grid-template-columns: 2fr 1fr;
}
body.page-about .about-split--image-first {
  grid-template-columns: 1fr 2fr;
}
```

**[FLAG → recommendation]** D-05 marks `{{< split >}}` as one of three shortcodes. **Whether it ships at all in Phase 9** depends on whether the rewritten `index.md` actually has split text+image rows (vs. flowing prose without them). Phase 9's content rebalance demotes climbing to "Outside Work" and cards Experience — neither of those obviously needs a split. **Recommendation: ship `{{< split >}}` anyway** — it satisfies REQ ABOUT-01 ("alternating ratios per section") more literally than role cards alone, and preserves design slack for the planner. If the planner concludes `index.md` has no split row, the shortcode goes unused and can be removed in a follow-up.

**Mobile reflow (REQ ABOUT-07, D-10):** Both variants collapse to `grid-template-columns: 1fr` at `< 600 px`. Source order preserved (markdown writes text-first or image-first; mobile reads top-down in source order). Pitfall 20 mitigation: any split where mobile must show image-first should declare `--image-first` explicitly so the markdown source order matches mobile reading order.

**Recommendation rationale for new arm parameters:**
- **`split` uses `fit` not `fill`** (Pitfall 19 mitigation): `fit` caps the longest edge at 600 px and preserves the source aspect ratio — a portrait stays portrait, a landscape stays landscape. The split row's CSS `1fr` slot then sizes the rendered image to the column width while `aspect-ratio` is intrinsic. `Smart` cropping is unnecessary because there is no crop. This avoids the "face clipped on mobile but not desktop" failure mode.
- **`feature` uses `fill 1024x576` (16:9)**: full-bleed banner-style image at 16:9 — the only ratio the design uses for full-bleed. `Smart` anchor handles entropy-based crop (Hugo 0.157.0 does not have face detection — Pitfall 19 verification gate: inspect every processed `feature` photo in build output before merge). 1024 px is enough for 2× retina at the site's 640 px content column.

**Hook implementation pattern (matches Phase 7 precedent — additive only):**
```go-html-template

{{- if $resource -}}

  {{- $isGrid    := eq $title "grid" -}}
  {{- $isSplit   := eq $title "split" -}}

  {{- if $isHero -}}
    {{- $processed = $resource.Process "fill 480x600 Smart webp q80" -}}
  {{- else if $isGrid -}}
    {{- $processed = $resource.Process "fill 400x300 Smart webp q75" -}}
  {{- else if $isSplit -}}
    {{- $processed = $resource.Process "fit 600x450 webp q78" -}}
  {{- else if $isFeature -}}
    {{- $processed = $resource.Process "fill 1024x576 Smart webp q80" -}}
  {{- else -}}
    {{- $processed = $resource.Process "fill 800x600 Smart webp q78" -}}
  {{- end -}}
  <img src="{{ $processed.RelPermalink }}"
       width="{{ $processed.Width }}"
```
